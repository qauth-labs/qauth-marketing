# design-sync notes — @qauth/ui

## Build recipe (no single buildCmd — multi-step)

1. `@qauth/ui` has no build target, explicit or Nx-inferred (confirmed via
   `bunx nx show project @qauth/ui --json` — only `lint` exists; `@nx/vite/plugin`
   only infers a `build` target for projects with their own `vite.config.*`, and
   `libs/ui` has none). The app consumes it as raw TS source via the
   `@qauth/ui` → `libs/ui/src/index.ts` path alias in `tsconfig.base.json`.
   `cfg.entry` is set to `libs/ui/src/index.ts` directly (the real barrel file,
   not a synthesized `export *` entry) — this is more faithful than synth-entry
   fallback because `index.ts` does selective named re-exports per file, not
   blanket `export *`.
2. Tailwind v4 needs its own compile step — CSS is **not** shipped as source,
   it's generated by scanning `@source`-listed directories for classes actually
   used. The app's own build (`@tailwindcss/vite`) only scans
   `apps/qauth-landing/src` + `libs/ui/src`, so a straight copy of the app's
   compiled CSS is a **closed set**: any utility class used only in
   `.design-sync/previews/**` (which four independent authoring passes hit
   repeatedly — `w-80`, `w-96`, `rounded-md`, `truncate`, `justify-end`,
   `grid-cols-3`, `divide-y`, `max-w-md`, arbitrary values, …) silently renders
   as a no-op: no error, no warning, the element just doesn't get that
   property. This already cost the first-graded calibration set (`Card`/`Field`
   both used `w-80` and rendered full-width instead of 320px, and still graded
   "good" — that's exactly why it's dangerous).

   Fix: a **design-sync-owned Tailwind entry**,
   `apps/qauth-landing/.ds-tailwind-entry.css` (committed — it's build-recipe
   source, not a generated artifact). It mirrors the real `src/styles.css`
   tokens/theme **verbatim** (keep them in sync by hand if the app's palette
   changes — no automated check for drift), adds `@source` for
   `.design-sync/previews` on top of the app + `libs/ui/src` sources the real
   file already scans, and carries a small `@source inline(...)` safelist for
   common width/spacing/grid utilities so a future preview author reaching for
   an unused-but-standard Tailwind class doesn't silently no-op again. It
   deliberately omits the app's page-decoration CSS (noise texture, grid
   lines) — that's marketing-page chrome, not component styling. It sits in
   `apps/qauth-landing/` (not `libs/ui/`) only because that's where `shadcn`
   and `tw-animate-css` resolve from (app-only devDependencies); it is never
   imported by the real app build.

   Compile it with the Tailwind CLI (pinned `@tailwindcss/cli@4.1.18` in
   `.ds-sync/node_modules`, matching the app's `tailwindcss@4.1.18` — reinstall
   if the app's version drifts):
   ```
   node .ds-sync/node_modules/.bin/tailwindcss \
     -i apps/qauth-landing/.ds-tailwind-entry.css \
     -o libs/ui/.ds-cache/compiled-tailwind.css
   ```
   `cfg.cssEntry` points at that output. It MUST live inside `libs/ui/` —
   `cssEntry` is PKG_DIR-bounded (uploaded verbatim, so the converter refuses
   paths outside the package), unlike `docsDir`/`extraFonts`/`tsconfig` which
   are workspaceRoot-bounded and can point at sibling dirs.

   Before relying on a class in a new preview, verify it survives the scan:
   `grep '\.<class-name> {' libs/ui/.ds-cache/compiled-tailwind.css` (note the
   space before `{` — Tailwind v4's output isn't minified this way).
3. Full build command:
   ```
   node .ds-sync/package-build.mjs --config .design-sync/config.json \
     --node-modules libs/ui/node_modules --out ./ds-bundle
   ```
   (`--entry` no longer needed on the CLI — it's in config now.)

## Component count is 293, not ~53

The repo has 53 top-level component *files*, but every sub-part export
(`AccordionContent`, `CardHeader`, `DialogTrigger`, etc.) is its own
component/card — expected converter behavior, not a bug. The core-subset
authoring scope (~28 primary names) is unaffected; the other ~265 ship as
fully-functional floor cards, authorable incrementally on future re-syncs.

## Fonts

`styles.css`'s own compiled `@font-face` rules use root-absolute `/assets/...`
paths (Vite production convention) that don't resolve as local files — 11 get
dropped as dead by the build. Real fonts come from `cfg.extraFonts`, pointed
at the `@fontsource-*` packages in `apps/qauth-landing/node_modules/` (DM Sans
Variable, Instrument Serif 400/400-italic, JetBrains Mono Variable) — 34
`@font-face` rules land correctly via that path.

## Overlay/portal components need `cardMode: single` (cfg.overrides)

Radix portals every overlay (`DialogContent`, `AlertDialogContent`,
`SheetContent`, `PopoverContent`, `TooltipContent`, `DropdownMenuContent`,
`SelectContent`) to `document.body`, so in a component's **default grid card**
(all cells shown together, no `?story=` param, `.ds-cell` has
`overflow:hidden`) an open overlay escapes its cell and paints over the whole
card instead of staying contained. **A clean capture/grade sheet does NOT
catch this** — `package-capture.mjs` drives `?story=<Label>`, which is already
a single full-bleed render, so the bug is invisible there. It only shows up in
the actual default (unscoped) `<Name>.html`. Confirmed and fixed via
`cfg.overrides` for `Dialog`, `AlertDialog`, `Sheet`, `Popover`, `Tooltip`,
`DropdownMenu`, `Select` (see config.json). Any future component whose story
forces an overlay open (`Command`, `Combobox`, `HoverCard`, `Drawer`, …) needs
the same treatment — check the default grid HTML, not just the capture sheet.

Two supporting findings from the same investigation:
- `Dialog`/`AlertDialog`/`Sheet` gate their desktop layout behind Tailwind's
  `sm:` breakpoint (measured against the card's own viewport) — **card
  viewport width must be ≥640px** or they render their mobile/stacked layout
  instead of the canonical desktop shape. `Popover`/`Tooltip`/`DropdownMenu`/
  `Select` have no `sm:` classes in their source, so no such floor applies to
  them.
- Prefer a controlled `open` (not `defaultOpen`) with no `onOpenChange` for
  the primary story of an overlay you're forcing open — `defaultOpen` is
  closable (Esc / click-outside / close button all still work), so the
  *live* product card could blank to a bare trigger. `open` is inert.
  Also add `onOpenAutoFocus={(e) => e.preventDefault()}` on the `*Content`
  piece if it contains a focusable with a `defaultValue` — Radix's
  autofocus-on-open otherwise paints a blue OS text-selection highlight over
  the value in the screenshot.

## `@tabler/icons-react` MUST be imported per-icon, never from the barrel

`import { IconFoo } from '@tabler/icons-react'` compiles in ~100+ **seconds**
per preview file, not milliseconds. Root cause: the package's barrel
(`dist/esm/tabler-icons-react.mjs`) opens with
`import * as index from './icons/index.mjs'; export { index as icons };` —
a namespace re-export of all ~5,989 icons that esbuild can't tree-shake away,
so bundling touches every one of those ~6,000 modules regardless of which 1-2
icons you actually use. That alone is fast in raw esbuild (~150ms, verified),
but `.ds-sync/lib/story-imports.mjs`'s custom import-policy resolver
(`ds-import-policy`, the `filter: /.*/` catch-all) does an async re-entrant
`b.resolve()` + `realpathSync` + regex/lookup for **every** resolution esbuild
asks it to arbitrate — paid ~6,000 times instead of ~2-3. That's what turned
28 authored previews into a 30+ minute hang instead of a few seconds. This is
a latent perf bug in the shared converter script (any DS pairing a barrel-style
icon/asset package with a custom resolver policy would hit it), not something
wrong with this repo — but the fix here is on the authoring side, not a lib
fork: import each icon from its own module:

```tsx
// NOT this — resolves the entire ~6,000-module barrel:
import { IconAlertTriangle } from '@tabler/icons-react'
// This — resolves exactly one file:
import IconAlertTriangle from '@tabler/icons-react/dist/esm/icons/IconAlertTriangle.mjs'
```

No `package.json` `exports` map restricts the subpath, so this resolves
cleanly. All 18 previews that used the barrel form were fixed this way at
close-out (verified: full 28-preview build went from 30+ min to <10s).
**Any future preview author reaching for a tabler icon must use the deep
import form** — this belongs in the conventions header too.

## Conventions worth carrying into the conventions header

- `Alert`'s icon-grid layout (`has-[>svg]:grid-cols-[auto_1fr]`) only fires
  when the icon is a **direct child** of `<Alert>`, siblings with
  `AlertTitle`/`AlertDescription` — not wrapped in a div.
- `Badge` icons need `data-icon="inline-start"` / `"inline-end"` on the icon
  element for correct asymmetric padding.
- `InputGroup` block addons (`align="block-start"` / `"block-end"`) only get
  their divider *and* correct padding with an explicit `className="border-b"`
  / `"border-t"` — the cva keys padding off the border class itself.
- `InputGroup`'s invalid state goes on the control (`InputGroupInput
  aria-invalid="true"`), not the `InputGroup` wrapper — the wrapper picks it
  up via a `has-[[data-slot][aria-invalid=true]]` selector.
- `Select`: pass `position="popper"` on `SelectContent` for stories that force
  it open — the default `item-aligned` overlays content on top of the
  trigger and can clip off the top of the viewport.
- Accordion triggers render in **Instrument Serif**, not DM Sans — Radix
  wraps `AccordionTrigger` in an `<h3>`, and the site styles headings with
  the display serif. This is faithful real behavior, not a token failure.
- `AvatarGroup` at default (32px) size clips 2-letter initials under the
  `-space-x-2`/`ring-2` overlap — use `size="lg"` (40px) in group contexts;
  `AvatarGroupCount` already has a `group-has-data-[size=lg]` rule to match.
- `AvatarImage`: point `src` at a repo-relative path for previews (e.g.
  `/team/name.jpg`) so it 404s instantly and Radix degrades to
  `AvatarFallback` — a remote URL risks hanging the capture's `networkidle`
  wait up to its per-cell timeout.

## Known render warns (triaged as legitimate)

- `InputOTPSeparator` — `[RENDER_THIN]`: renders `<IconMinus/>` only (no text),
  the floor card's thin-content heuristic flags it. Not authored (out of the
  core-subset scope this run). Nothing broken — just an icon-only separator.
- `BreadcrumbEllipsis`, `PaginationEllipsis`, `SidebarMenuSkeleton`, `Slider` —
  `[RENDER_BLANK]`: floor card's crash-prevention props render near-blank
  (<5KB PNG) for these. All out of core-subset scope this run — authorable
  incrementally on a future re-sync via `.design-sync/previews/<Name>.tsx`.
  Not broken, just unauthored.

## Known repo issues (pre-existing, not design-sync's to fix)

- `Calendar` (`calendar.tsx`) uses `rounded-(--cell-radius)` throughout, but
  `--cell-radius` is never defined anywhere — not in `styles.css` tokens, not
  set at runtime via inline style. `[TOKENS_MISSING]` flagged it. This looks
  like a genuine pre-existing bug in the app's own component (day cells likely
  render with no border-radius) — not something to invent a value for here.
- `--radix-accordion-content-height`, `--radix-navigation-menu-viewport-height`,
  `--radix-navigation-menu-viewport-width` also flagged by `[TOKENS_MISSING]` —
  these are Radix's own runtime-set animation vars (JS sets them via inline
  style). Expected to be absent from static stylesheets — non-issue.
- `--tw` flagged by `[TOKENS_MISSING]` — looks like a scraper false-positive
  from an arbitrary-value Tailwind selector, not a real missing token.

## Re-sync risks

- `libs/ui/.ds-cache/compiled-tailwind.css` is a manual snapshot (gitignored).
  It MUST be regenerated (step 2 above) before every build/re-sync, or the
  synced DS will silently drift from the app's real compiled styles.
- `apps/qauth-landing/.ds-tailwind-entry.css`'s `:root`/`.dark`/`@theme` blocks
  are a **hand-maintained copy** of `apps/qauth-landing/src/styles.css`'s
  tokens — nothing checks them for drift. If the app's palette/theme changes,
  this file needs the same edit or the synced DS will render stale tokens.
  Worth a scripted diff check on a future re-sync.
- `cfg.extraFonts` points into `apps/qauth-landing/node_modules/@fontsource-*`.
  If those packages are removed, renamed, or restructured, re-sync needs
  `cfg.extraFonts` updated to match.
- Playwright pinned to 1.58.0 in `.ds-sync/node_modules` to match the machine's
  cached chromium build (revision 1208) and avoid a ~200MB download. A fresh
  clone / different machine may need a different pinned version — check
  `~/.cache/ms-playwright/` and match via
  `https://raw.githubusercontent.com/microsoft/playwright/v<X.Y.Z>/packages/playwright-core/browsers.json`.
